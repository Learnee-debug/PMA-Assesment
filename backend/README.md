# Weather App — Backend (Tech Assessment #2, Python)

FastAPI + SQLite REST API providing CRUD on weather records for a date range, plus
data export and third-party integrations (YouTube, OpenStreetMap).

## Requirements
- Python 3.10+

## Setup & Run
```bash
cd backend
python -m venv venv
# Windows:
venv\Scripts\activate
# macOS/Linux:
source venv/bin/activate

pip install -r requirements.txt
python run.py          # listens on http://localhost:4000
```
The SQLite database file `weather.db` is created automatically in `backend/` on first run.
Interactive API docs are auto-generated by FastAPI at http://localhost:4000/docs.

## Design notes / what this demonstrates
- **No API key required.** Geocoding + weather come from [Open-Meteo](https://open-meteo.com/),
  which is free and keyless — easier for a take-home grader to run as-is.
- **Location input flexibility**: city/town/landmark/zip names go through Open-Meteo's
  geocoding (fuzzy "best match" on the top result); `"lat,lon"` strings are detected and
  used directly as GPS coordinates (`app/weather_api.py`).
- **CRUD** lives in `app/main.py` against the `records` table (`app/db.py`, raw `sqlite3`):
  - `POST /api/records` — create (validates date range + resolves/validates location, stores resolved weather)
  - `GET /api/records` / `GET /api/records/{id}` — read
  - `PUT /api/records/{id}` — update (re-validates and re-fetches weather if location/dates change)
  - `DELETE /api/records/{id}` — delete
- **Export**: `GET /api/records/export/{format}` for `json`, `csv`, `xml`, `md`, `pdf`
  (`app/exporters.py`, using `csv`, `dicttoxml`, and `fpdf2`).
- **Integrations**: `GET /api/integrations/youtube?location=` (search link) and
  `GET /api/integrations/map?lat=&lon=` (OpenStreetMap embed, no API key needed).
- **Live weather (used by the frontend widget)**:
  `GET /api/weather/lookup?location=` and `GET /api/weather/by-coords?lat=&lon=`
  return current conditions + 5-day forecast.
- **Validation** (`app/validators.py`): date-range format/order/max-span checks, and
  location presence/length checks before hitting the geocoder.

## API Quick Reference

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/weather/lookup?location=` | Current weather + 5-day forecast for a free-text location |
| GET | `/api/weather/by-coords?lat=&lon=` | Same, for raw coordinates (used by "Use my location") |
| POST | `/api/records` | Create a saved record `{ location, startDate, endDate }` |
| GET | `/api/records` | List all saved records |
| GET | `/api/records/{id}` | Get one record |
| PUT | `/api/records/{id}` | Update a record `{ location?, startDate?, endDate? }` |
| DELETE | `/api/records/{id}` | Delete a record |
| GET | `/api/records/export/{format}` | Export all records (`json`\|`csv`\|`xml`\|`md`\|`pdf`) |
| GET | `/api/integrations/youtube?location=` | YouTube search URL for the location |
| GET | `/api/integrations/map?lat=&lon=` | OpenStreetMap embed/view URLs |

All error responses are `{ "errors": ["..."] }` with a 4xx/5xx status — this matches the
shape the frontend (`frontend/src/api.js`) already expects.
